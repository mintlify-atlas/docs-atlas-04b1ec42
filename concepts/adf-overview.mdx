Auto-lead Data Format (ADF) 1.0 is an XML-based standard that defines how automotive purchase leads are structured and exchanged between systems. adf-kit is a TypeScript library that lets you parse inbound ADF documents, generate outbound leads, validate them against the spec, and round-trip between XML and typed objects — all without runtime dependencies.

## Background and adoption

ADF was developed to give the automotive industry a common language for transmitting lead data. Before ADF, each CRM, dealer management system (DMS), and lead aggregator used proprietary formats, requiring custom integrations for every connection. ADF 1.0 standardised the envelope so that a lead generated by AutoTrader, Cars.com, or a dealer's own website could be consumed by any ADF-aware CRM without transformation.

Today, ADF 1.0 is the widely-used version of the spec and is supported by virtually every major participant in the automotive lead ecosystem:

- **Lead aggregators** — AutoTrader, Cars.com, TrueCar, Edmunds
- **CRMs** — DealerSocket, Reynolds & Reynolds, VinSolutions
- **DMS platforms** — CDK Global, Reynolds ERA
- **Dealer websites** — OEM portals, inventory management systems

## Document structure

An ADF document is an XML envelope (`<adf>`) containing one or more `<prospect>` elements. Each prospect is a single lead — one buyer expressing interest in one or more vehicles at one dealer. The hierarchy looks like this:

```text
ADFDocument
└── prospects: ADFProspect[]
    ├── requestDate: ISO8601DateTime
    ├── status: 'new' | 'resend'
    ├── ids: ADFId[]
    ├── vehicles: ADFVehicle[]
    │   ├── year, make, model, vin, stock, trim, bodyStyle …
    │   ├── interest: 'buy' | 'lease' | 'sell' | 'trade-in' | 'test-drive'
    │   ├── status: 'new' | 'used'
    │   ├── condition: 'excellent' | 'good' | 'fair' | 'poor' | 'unknown'
    │   ├── odometer: ADFOdometer  { value, units: 'mi'|'km', status }
    │   ├── price: ADFPrice        { value, type, currency, delta, relativeTo }
    │   ├── colorCombinations: ADFColorCombination[]
    │   ├── options: ADFOption[]
    │   └── finance: ADFFinance    { method, amounts, balance }
    ├── customer: ADFCustomer
    │   ├── contact: ADFContact    (names, emails, phones, address)
    │   └── timeFrame: ADFTimeFrame { description, earliestDate, latestDate }
    ├── vendor: ADFVendor
    │   ├── vendorName: string
    │   └── contact: ADFContact
    └── provider?: ADFProvider
        └── name, service, url, email, phone, contact
```

Every ADF document has an ADF version attribute (currently `"1.0"`). The `<prospect>` element is the atomic unit — it captures the buyer, the vehicle(s) of interest, the receiving dealer, and optionally the lead source (provider).

## ADF version

The current widely-adopted version of the standard is **ADF 1.0**. adf-kit targets ADF 1.0 exclusively and sets `version="1.0"` on all generated documents. The spec is published by the ADF working group and is available at [adfxml.info/adf_spec.pdf](https://adfxml.info/adf_spec.pdf).

## How adf-kit fits in

adf-kit provides four capabilities that cover the full lead lifecycle:

| Function | Description |
|---|---|
| `ADFparse(xml)` | Parse an ADF XML string into a typed `ADFDocument` |
| `ADFvalidate(doc)` | Validate against the ADF 1.0 spec — returns errors and warnings without throwing |
| `ADFserialize(doc)` | Serialize a typed `ADFDocument` back to ADF XML |
| `ADFProspectBuilder` | Fluent builder for a single lead — call `.string()` or `.xml()` to get output |
| `ADFBuilder` | Fluent builder for a multi-prospect document |
| `generateProspect(input)` | One-shot factory: convert a `ProspectInput` object directly to XML |
| `generateADF(input)` | One-shot factory: convert an `ADFInput` object directly to XML |

A typical inbound integration uses `ADFparse` and `ADFvalidate`. A typical outbound integration uses `ADFProspectBuilder` or `generateProspect`. Round-trip transformations use all four in sequence.

<Info>
  The full ADF 1.0 specification is published at [adfxml.info/adf_spec.pdf](https://adfxml.info/adf_spec.pdf). adf-kit validates documents against this spec via `ADFvalidate`.
</Info>
